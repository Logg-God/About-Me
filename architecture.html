Architecture

Technical Architecture

A modular service-based structure designed for scaling and collaboration.

Service-based architecture

The project uses a service pattern: each system owns a single responsibility and exposes a clean API. This keeps the codebase organized and makes it easy for multiple contributors to work safely.

Core services

  • CombatService: validates hits, applies damage, handles cooldowns (server-authoritative)
  • EnemyService: spawns enemies, runs AI, tracks enemy health/death
  • DungeonService: controls run flow: rooms, objectives, doors, boss triggers
  • LootService: drop tables, rarity rolls, reward generation
  • DataService: save/load coins and unlocks (later persistence)

Client vs Server responsibilities

  • Client: input, animations, UI, camera feedback
  • Server: validation, damage, enemy health, rewards, progression

Why server-authoritative combat

Clients can be exploited. The server should be the source of truth for damage and rewards. The client requests actions; the server validates them and applies results.

Collaboration workflow

  • GitHub repo with branches: main (stable), dev (work-in-progress)
  • Feature branches for contractors: ui-polish, vfx-pass
  • PR-style reviews (my dad can review changes before merge)
